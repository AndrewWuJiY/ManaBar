## 1. 技术栈

| 层 | 选择 | 备注 |
|---|---|---|
| 语言 | Swift 6 | 严格并发 |
| 工程 | Xcode `ccbar.xcodeproj` | 单 target，标准 macOS App |
| UI | SwiftUI 主、AppKit 辅 | 菜单栏 label / 悬浮窗 panel 走 AppKit |
| 最低系统 | macOS 14 Sonoma+ | 依赖 `MenuBarExtra(.window)`、`@Observable`、`SMAppService` |
| 存储 | UserDefaults + JSON 文件 | 不引入 SQLite |
| 网络 | `URLSession` async/await | 标准库，无第三方依赖 |
| 图表 | `Charts`（SwiftUI） | 用于统计 tab 日趋势 |

整个工程零第三方依赖；钥匙串通过命令行 `security` 访问，避免 Keychain Services API 的多次授权弹窗。

## 3. 应用入口与窗口组合

`CCBarApp.swift` 组装三个 Scene：

1. `MenuBarExtra(.window)`：label 是自绘 NSImage 包装；content 是 `PopoverRootView`。`.task` 内执行 `appState.bootstrap()` 并把 `FloatingPanelController` 与 AppState 绑定，再根据设置同步显示。
2. `Window("CCBar", id: "main")`：主窗口，内部 `MainWindowRootView` 用 TabView 切「统计 / 设置」。
3. `Window("Welcome", id: "onboarding")`：引导窗口，`fixedSize` 禁缩放。

`AppCommands` 通过 `CommandGroup` 注入全局快捷键：

- `⌘,` → 打开主窗口并切到设置 tab。
- `⌘1` → 打开主窗口并切到统计 tab。
- `⌘R` → 调用 `appState.refreshNow()`。

`AppDelegate` 只做一件事：`NSApp.setActivationPolicy(.accessory)` 让进程不出现在 Dock。

## 4. 全局状态：`AppState`

`@Observable @MainActor`，单例方式由 `CCBarApp` 持有并通过 `.environment` 注入。核心字段：

- 凭据：`codexAccount` / `claudeAccount`，加载错误 `codexError` / `claudeError`。
- 额度：`codexQuota` / `claudeQuota`（`QuotaSnapshot`），来源 `codexQuotaSource` / `claudeQuotaSource`（`api` / `cache` / `cliFallback`）。
- 刷新状态：`QuotaRefreshState`（`lastSuccessAt`、`lastAttemptAt`、`backoffUntil`、`lastError`、`inFlight`、`source`）。
- 今日花费：`codexTodayCost` / `claudeTodayCost`，由 `UsageService.publishTotals()` 写入。
- UI 协调：`mainTab`（统计 / 设置）、`shouldShowOnboarding`。
- 子系统：`usageService`、`scheduler`、`quotaCache` 内存副本、Claude CLI 兜底冷却 `claudeFallbackBackoffUntil`。

关键方法：

- `bootstrap()`：只执行一次；按顺序加载 quota 缓存 → `UsageService.bootstrap` → 读 Codex 凭据 → 必要时弹 Keychain 引导 → 读 Claude 凭据 → 决定是否展示引导 → 启动 Scheduler → 触发首次 JSONL 扫描（异步不阻塞）。
- `refreshNow()`：用户手动刷新；触发一次额度刷新、JSONL 扫描和服务状态拉取。
- `refreshQuotas(reason:)`：每次刷新(手动 / Scheduler 定时)都先重读 Codex / Claude 凭据，再分别调度额度刷新并写日志摘要。`loadCodex` / `loadClaude` 内部比较 `accountId` / `email`，发现身份变化(例如 cc-switch 外部切换了账号)时清掉旧的额度快照与磁盘缓存，避免出现"新账号 + 旧额度"的错配。
- `applySettingsChange()`：设置项变化后立即把新间隔灌进 Scheduler。

刷新前置校验在 `beginCodexRefresh` / `beginClaudeRefresh` 中：检查 `inFlight`、`backoffUntil`、周期性触发的 60s 最小间隔。成功后 `storeXxx` 写入快照 + 持久化；失败走 `markXxxFailure`，遇 429 设置 10 分钟退避。

## 5. 凭据读取

### 5.1 Codex

`CodexAuth.load()` 读 `~/.codex/auth.json`，解析 `tokens.access_token` / `refresh_token`，再用 `JWT.decode` 解 `id_token` 的 payload 拿邮箱、`chatgpt_plan_type`、`account_id`。

`CodexTokenRefresher.ensureFreshAccessToken`：当 JWT `exp` 剩余 < 300s 时 POST 到 OpenAI OAuth token 端点（grant_type=refresh_token，client_id 用 Codex CLI 公开 ID），成功后原子写回 `auth.json`。

### 5.2 Claude

`ClaudeAuth.load()` 优先读 `~/.claude/.credentials.json`；不存在再调 `security find-generic-password -s "Claude Code-credentials" -w` 读 Keychain。邮箱兜底来自 `~/.claude.json` 顶层 `oauthAccount.emailAddress`。

`ClaudeTokenRefresher.ensureFreshAccessToken`：

- 端点 `https://platform.claude.com/v1/oauth/token`，client_id 用 Claude Code CLI 的公开 ID。
- 续期成功后按原存储位置（`.file` / `.keychain`）回写。Keychain 回写先用 `security find-generic-password` 读出原 JSON，merge token 字段后用 `security add-generic-password -U -s <service> -a $USER -w <json>` 整体覆盖，保留 `subscriptionType` 等附加字段。
- `expiresAt` 用毫秒时间戳与 CLI 对齐。

#### 与其他客户端共存的刷新策略

Claude 的 OAuth `refresh_token` 是 *一次性* 的：每次刷新服务端会旋转一个新值，旧值立即作废。用户机器上同时跑 Claude Desktop / Claude Code CLI / cc-switch / codexbar / cc-bar 时，谁先刷新成功，其他人手里的 `refresh_token` 都会变成 `invalid_grant`。为此 cc-bar 把自己当 *只读监控* 设计成"被动优先"：

1. **缩短 skew 让出主动权**：`refreshSkew` 由 300s 收紧到 30s，把 access_token 临期的主动刷新让给 CLI / Desktop。
2. **进程内串行 + 去重**：`Coordinator` actor 维护 `inFlight` Task，菜单栏 / 悬浮窗 / Popover 三个入口几乎同时触发时合并成一次刷新，不会自家跟自家争抢。
3. **拿锁后先重读存储**：真正发请求前 `peekStored(source:)` 再读一次 credentials.json / Keychain；若已被别的客户端写入新鲜 access_token，直接采用，不发请求；若 refresh_token 已被旋转，至少用最新一份去发请求。
4. **文件 mtime 礼让窗口**：`~/.claude/.credentials.json` 在最近 `politeMtimeWindow=10s` 内被改过，认为别人刚完成刷新，sleep 1s 再重读一次。
5. **`invalid_grant` 软恢复**：拿到 `HTTP 400 invalid_grant` 不立即报错——sleep 1s 再重读，若 refresh_token 已被外部更新且 access_token 仍新鲜，静默采用、对 UI 报告成功。
6. **委托 CLI 兜底刷新**：软恢复仍失败时,`ClaudeDelegatedRefresh` 会尝试唤起本机 `claude` CLI(优先 `$CLAUDE_CLI_PATH` → 常见 PATH 位置 → Claude Desktop 内嵌的 `~/Library/Application Support/Claude/claude-code/<ver>/claude.app/Contents/MacOS/claude`),在最小 PTY 里跑一次 `/status`,让 CLI 用受信任的会话身份完成刷新并把新凭据写回 keychain / credentials.json;通过对存储指纹比对来判断是否成功。带 5 分钟冷却 + in-flight 去重,避免反复唤起。成功则静默使用新 token;CLI 找不到 / 仍失败时才升级成 `QuotaError.tokenRevoked`,给出"请运行 `claude` 重新登录"的友好提示。
   - 设计意图:在用户没主动操作 / 只在 Claude Desktop 中登录的常见场景下尽量自动恢复,而 Desktop 自身不会回写 CLI 的 `Claude Code-credentials` keychain 条目,所以需要 CLI 做这次代办。
   - **TCC 隔离**:为了避免 claude CLI 启动时扫描用户 home 目录触发"CCBar 想访问 文稿 / 桌面 / 相册..."这种归因到主体 App 的弹窗,实现做了两层防护:
     * **受控工作目录** `~/Library/Application Support/CCBar/ClaudeProbe/`:claude 启动时 cwd 设到这里,里面只有空的 `.claude/settings.local.json`(关掉 deep-link 注册),它扫不到任何用户项目结构。
     * **Watchdog 套娃** `CCBar.app/Contents/Helpers/CCBarClaudeWatchdog`:一个 ~120 行的小 Swift 二进制([Helpers/CCBarClaudeWatchdog/main.swift](../Helpers/CCBarClaudeWatchdog/main.swift)),只做 `posix_spawnp` claude + 信号转发 + 父进程死亡守护。cc-bar 通过 watchdog 启动 claude,macOS TCC 沿进程链上溯时看到的是独立签名的 watchdog 而非主体 App,任何潜在的权限弹窗都归因到 helper,不污染 cc-bar 的权限画像(对齐 CodexBar 的 `CodexBarClaudeWatchdog` 做法)。
     * 构建方式:CCBar target 加了一个 `Embed CCBarClaudeWatchdog` 的 Shell Script Build Phase(在 Resources 之后),用 `swiftc` 编译 main.swift 并 codesign 到 `Contents/Helpers/CCBarClaudeWatchdog`。Debug 跑(尤其 Xcode Run 但 bundle 里没 helper)会降级到直接起 claude,可能弹一次权限框,可以接受。

> Codex 那边目前还是原始策略（`refreshSkew=300`，无 Coordinator / 软恢复 / 委托刷新），后续按需对齐。

## 6. 额度查询

### 6.1 接口

| 服务 | 请求 | 关键 Header |
|---|---|---|
| Codex | `GET https://chatgpt.com/backend-api/wham/usage` | `Authorization: Bearer …`、`ChatGPT-Account-Id`、`User-Agent: codex-cli` |
| Claude | `GET https://api.anthropic.com/api/oauth/usage` | `Authorization: Bearer …`、`anthropic-beta: oauth-2025-04-20`、`User-Agent: claude-code/2.1.0` |

两边响应都映射到统一的 `QuotaSnapshot { app, fiveHour, weekly, weeklyOpus?, weeklySonnet?, planType?, fetchedAt }`。`QuotaWindow.usedPercent` 来自原始字段（Codex 是 `used_percent`，Claude 是 `utilization`），`remainingPercent = 100 − usedPercent`。

### 6.2 调用策略

- 启动只读 `quota-cache.json` 灌进 AppState，**不**立刻打网络。
- 周期循环按设置间隔触发（默认 5 分钟，可关闭）；任意一家上次成功 < 60s 时跳过。
- 收到 429 后该服务进入 10 分钟退避，期间所有触发短路。
- 手动刷新绕过 60s 节流，但仍遵守 429 退避。
- 401 / 403 不会自动停止循环（保持简单），但错误会显示在 Popover。
- 网络/解析失败保留旧快照，错误写入 `lastError`，下一 tick 重试。

### 6.3 Claude CLI 兜底

`ClaudeCLIFallbackQuotaClient` 仅在 `refreshNow` 触发 Claude API 失败 **且** `claudeQuota == nil` 时启动，且自身有 10 分钟独立冷却：

- 候选可执行：`/opt/homebrew/bin/claude` → `/usr/local/bin/claude` → `/usr/bin/claude` → `env claude`。
- 用 `/usr/bin/script -q /dev/null <claude>` 包一层伪 PTY 拿到 ANSI 输出；失败再裸跑一次直接执行。
- 通过 stdin 写入 `/usage\n\n/status\n/exit\n` 驱动 CLI。
- 输出 strip ANSI，按 `Current session` / `Current week (all models)` / `Current week (Sonnet only)` / `Current week (Opus)` 标签抓百分比与 `Resets …` 文本。
- 错误关键词识别：`rate_limit_error` / `authentication_error` / `failed to load usage data`，转成对应 `QuotaError`。

CLI 兜底拿到的 snapshot 用 `QuotaSnapshotSource.cliFallback` 标记，仅用于救急展示。

### 6.4 持久化

`QuotaCache` 写 `~/Library/Application Support/CCBar/quota-cache.json`，结构 `{ codex: Record?, claude: Record? }`，每条记录含 snapshot、source、updatedAt。

`QuotaHistoryStore` 写 `~/Library/Application Support/CCBar/quota-history.json`（payload v2），**跨天留存** 5H 额度变化历史，供时间线按区间回看：

- `lastSamples`：每个账号最近一次成功采样的 5H 剩余整数百分比，作为下一次对比基准；跨天保留以保证跨日 delta 连续。
- `events`：只有 5H 剩余整数百分比发生变化时才新增事件，字段包含账号 key、应用、账号类型、采样时间、变动前 / 后剩余、delta 和 5H 重置时间。
- 保留窗口 `retentionDays = 90`：启动和每次写入按 `retentionStart`（今天 0 点往前 89 天）裁剪 `events` / `lastSamples`，避免文件无限增长。
- 迁移：旧版 v1 只存当天的 `quota-history-today.json`，首次加载若新文件不存在会把其内容搬入新存储后按窗口裁剪。
- Codex 主账号、Claude 主账号、每个导入 Codex 账号分别使用独立 key；镜像主 Codex 的导入账号只在主账号来源为 API 成功快照时记录。

### 6.5 服务状态（`ServiceStatusClient`）

OpenAI / Anthropic 各自的 Statuspage.io 公开接口,无需鉴权,用于在 Popover 中显示一个状态圆点：

| 服务 | 端点 |
|---|---|
| OpenAI | `https://status.openai.com/api/v2/status.json` |
| Anthropic | `https://status.claude.com/api/v2/status.json` |

响应中 `status.indicator` 取值 `none / minor / major / critical / maintenance`，未识别值归并为 `unknown`。`ServiceStatusClient.fetch` 解析为 `ServiceStatus { indicator, description, updatedAt, fetchedAt }`，10s 超时。

调用策略：

- `AppState.refreshServiceStatus()` 并发拉两家，单边失败不影响另一边；网络/解析失败保留上一份快照，不清空。
- 由 `Scheduler` 固定 5 分钟一次驱动；启动后立即拉一次；`refreshNow()` 也会顺带刷新。
- 仅做展示，不缓存到磁盘。

## 7. 本地 JSONL 扫描

### 7.1 Claude（`ClaudeJSONLScanner`）

- 入口路径 `~/.claude/projects`，遍历所有 `.jsonl`。
- 每条 `type=assistant` 行解析 `message.id` / `message.model` / `message.usage`（`input_tokens` 不含 cache_read）/ `stop_reason`。
- 本批次内按 `message.id` 取唯一，优先选 `stop_reason != nil` 或更高 output_tokens 的一条；最后与全局 `seen` 集合比对，避免 sidechain / subagent 文件中重复计数。
- 全局 seen 列表持久化在 `scan-state.json`，上限 20000，超出截尾。

### 7.2 Codex（`CodexJSONLScanner`）

- 入口路径 `~/.codex/sessions` + `~/.codex/archived_sessions`。
- `turn_context` 行更新当前模型（写入 `ScanFileState.lastModel`）。
- `event_msg` 且 `payload.type == "token_count"` 行，从 `payload.info.last_token_usage` 直接累计（不是累计 delta），`info == nil` 跳过。
- `billable_input = input_tokens − cached_input_tokens`（Codex 的 `input_tokens` 含 cache_read）。

### 7.3 增量与持久化（`ScanCache`）

- 每个文件维护 `ScanFileState { mtime, offset, lastModel? }`。
- 调度逻辑：`mtime == prev && offset == size` → 跳过；`offset > size` → 文件被截断，归零重扫（去重靠 seen 集合）；否则从 `offset` 处续读到末尾。
- 已不存在的文件 watermark 在每次扫描末尾清理。
- 顶层结构 `ScanState { version, pricingFingerprint, claude, codex, claudeSeenMessageIds }`，版本号 v4。`UsageRollupPayload { version, pricingFingerprint, buckets, updatedAt }` 同步用 v4。
- **失效条件有二**：`version` 管「结构变更」（字段增减导致解码不兼容时 bump）；`pricingFingerprint`（`Pricing.fingerprint`，价格表内容的 SHA-256）管「价格变更」。load 时任一与当前值不一致即整份丢弃，触发全量重扫重算历史桶。改价格表（新增模型、改价、修正数值）后无需手动 bump 版本号——指纹自动失效，避免「改了价却忘了重算」导致历史 cost 不一致。

### 7.4 聚合（`UsageAggregator`）

- `@Observable @MainActor`，内存 `[BucketKey: UsageBucket]`，BucketKey = (day, app, model)。
- `ingest(entries)` 累加 token 与 cost；`snapshot()` 返回数组用于落盘；`todayCost(for:)` / `totals(app:from:to:)` 给 UI 用。
- 由于是 `@Observable`，Popover 和统计 tab 在 body 读 `aggregator.totals(...)` 时自动建立依赖，新扫描完写回 buckets 后 UI 自动刷新。
- **Token 展示口径**：`UsageTotals.inputWithCacheTokens = input + cache_read + cache_creation`，`totalTokens = inputWithCacheTokens + output`。Stats 页的"总令牌 / By service tokens / By model 入"统一使用此口径，与 cc-switch / ccusage 对齐；其中 Claude 的 `input_tokens` 本身不含 cache_read，Codex 的 `inputTokens` 在解析时已扣过 `cached_input_tokens`，因此三项相加不会重叠。

### 7.5 价格（`Pricing`）

- 表硬编码于 `Pricing.swift`，命中不到的 model cost = 0，token 仍记录。
- 归一化：剥 `openai/` 前缀；Vertex 风格 `name@YYYYMMDD` 截断 `@` 前；Anthropic 风格 `-YYYY-MM-DD` 或 `-YYYYMMDD` 后缀剥掉；最后 `lowercased()`。
- 金额用 `Decimal`（`Foundation.Decimal`），单位 USD / 百万 token。
- `Pricing.fingerprint`：价格表内容的 SHA-256，写进扫描状态 / 汇总缓存；表一变指纹即变，缓存自动失效触发全量重扫重算（见 §11 持久化失效条件）。

### 7.6 协调（`UsageService`）

- `bootstrap` 在 AppState 启动时读取 `usage-rollup.json` 灌进 aggregator，立即对外发布今日 cost。
- `scanNow()` 防重入：从 `ScanCache` 读旧 watermark，Claude / Codex 扫描在 detached `.utility` 优先级 Task 并行执行，`ingest` 后写 `scan-state.json` 与 `usage-rollup.json`，更新 `lastScanAt` 并 publish today cost。

## 8. 调度（`Scheduler`）

`@MainActor`，持有 AppState 弱引用，三条独立 `Task` 循环：

- **额度循环**：`quotaInterval` 秒触发一次 `appState.refreshQuotas(reason: .periodic)`。设置项可选 `1 / 2 / 3 / 5 / 10` 分钟，默认 2 分钟；`Scheduler` 仍保留对 `nil` / 0 的兜底，不启动循环。
- **用量循环**：`usageInterval` 秒触发一次 `appState.usageService.scanNow()`。
- **服务状态循环**：固定 5 分钟触发一次 `appState.refreshServiceStatus()`，与额度/用量解耦（statuspage 变化慢，无需跟随用户的额度刷新间隔）。

`setQuotaInterval` / `setUsageInterval` 在间隔变化时取消旧 task 并按新参数重启，对应设置变更的实时生效。

## 13. 已知技术债 / 持续关注

- **Claude CLI 兜底依赖外部进程**：用户没装 CLI 时会失败，目前只在 API 失败且无缓存的手动刷新触发，影响面有限。
- **价格表手维护**：模型更新需要同步 `Pricing.swift`；改表后由 `Pricing.fingerprint` 指纹自动失效缓存触发重扫，无需手动 bump 版本号。
- **seen ID 上限 20000**：极端长时间使用 + 大量 sidechain 引用可能导致环形丢失；目前足以覆盖正常使用。
- **没有自动更新通道**：发布要手动出包；后续可接 Sparkle。

## 14. 导入账号（其他 Codex 账号）

用户可在设置页手动粘贴 `~/.codex/auth.json` 内容，为 cc-bar 添加任意多个 Codex 副账号（仅查看配额，不影响 CLI 的当前登录状态）。

### 14.1 设计原则

- **凭据与默认账号解耦**：默认账号始终来自实时读取 `~/.codex/auth.json`（cc-switch 切号也跟着变）；导入账号是独立副本，各自维护 token，绝不写回 `~/.codex/auth.json`。若导入账号与当前默认账号身份相同，额度展示与刷新状态复用默认账号快照，不再读取导入 token 或重复请求官方接口；同时把当前默认账号 token 同步到该导入账号的 Keychain 副本，修复导入凭据长期未更新导致的后续刷新失败。
- **允许重复**：同一账号 ID 可以同时出现在默认槽和导入列表，语义不同（「当前 CLI 账号」vs「我关注的账号」），不强制去重；重复条目仅作为展示条目保留，刷新时镜像默认账号数据。
- **同 ID 覆盖**：再次粘贴相同账号时静默覆盖 token（保留 alias / visible 设置），避免悬空旧 token。
- **Team 账号区分用户**：账号身份用 `{chatgpt_account_id}:{chatgpt_user_id}` 复合键，同一 Team account 下不同 user 视为独立条目；缺失 `chatgpt_user_id` 时退化为单段 `chatgpt_account_id`（老数据兼容）。HTTP 请求里的 `ChatGPT-Account-Id` header 仍只取前半段。

### 14.2 存储

| 数据 | 位置 | 说明 |
|---|---|---|
| 元数据（id / alias / email / planType / visibleInPopover / addedAt） | `Application Support/CCBar/imported_codex_accounts.json` | 原子写，JSON version = 1。`id` 为 `{account_id}:{user_id}` 复合键 |
| Token 三件套（access / refresh / id） | macOS Keychain，`service = com.cc-bar.codex.imported`，`account = ImportedCodexAccount.id`（复合键） | SecItem 直接读写，不弹用户授权 |
| 额度快照缓存 | `quota-cache.json` 的 `importedCodex` 字段（`[String: QuotaCacheRecord]`） | 重启时立即显示上次数据 |

### 14.3 Token 续期回写隔离

`CodexTokenRefresher.ensureFreshAccessToken` 新增 `writeBack: WriteBack` 参数：
- `.codexAuthJSON`（默认）：续期后覆盖写回 `~/.codex/auth.json`，与现有行为一致。
- `.importedAccount(id:)`：续期后仅更新 Keychain 对应条目，不碰 `~/.codex/auth.json`。

### 14.4 刷新调度

`AppState.loadAllImportedCodexQuotas(reason:)` 在 `refreshQuotas` 中调用，对所有 `visibleInPopover = true` 的导入账号：
- 分批并发，每批最多 3 个同时 in-flight（chunked TaskGroup）。
- 每个账号独立维护 `QuotaRefreshState`（backoff / inFlight / lastSuccessAt），互不影响。
- 若导入账号与当前默认 Codex 账号匹配（优先 `{chatgpt_account_id}:{chatgpt_user_id}`，任一侧缺 user_id 时退回纯 account_id），直接镜像 `codexQuota / codexQuotaError / codexRefreshState`，不读取导入账号 Keychain token、不续期、不请求 `wham/usage`；若默认账号有 `access_token`，同步写入导入账号 Keychain，供该账号后续不再是默认账号时独立刷新。

### 14.5 排序

- 列表顺序即 `imported_codex_accounts.json` 中 `accounts` 数组顺序,Popover「其他账号」按相同顺序展示。
- 设置页支持拖动排序:每行 `.draggable(account.id)` + `.dropDestination(for: String.self)`,放下后调 `AppState.reorderImportedCodexAccounts(orderedIds:)` 写回 `ImportedCodexStore.saveAll`,再触发 `reloadImportedCodexAccounts` 同步运行时。
- 新增的账号追加到列表末尾;`reorderImportedCodexAccounts` 对未在传入 ids 中的条目自动追加到末尾,避免丢失。

### 14.6 Paste 解析（ImportedCodexPaste）

支持两种粘贴形态：
1. **完整 auth.json**（`{OPENAI_API_KEY, tokens: {...}, ...}`）
2. **裸 tokens 子对象**（`{access_token, refresh_token, ...}`）

解析优先级：`tokens.account_id` → `access_token` JWT payload → `id_token` JWT payload。
email / planType 从 `id_token` 解，缺 id_token 时再从 `access_token` 的 JWT 里找。
