# cc-bar Agent Guide

本文件同时给 Claude Code 和 Codex 使用。`AGENTS.md` 应作为指向本文件的软链接，避免两份规则不一致。

## 项目概况

cc-bar 是一个原生 macOS 菜单栏 App，用 Swift / SwiftUI 实现，用于展示 Codex 和 Claude Code 的额度、刷新状态、本地用量统计和后续桌面悬浮窗。

当前工程是 Xcode 工程：

- 入口：`CCBarApp.swift`
- 全局状态：`Core/AppState.swift`
- 菜单栏：`MenuBar/`
- 设置占位：`Settings/`
- 凭据 / 额度 / 调度 / 存储：`Core/`
- Xcode 工程：`ccbar.xcodeproj`

App 目前按里程碑逐步实现。开发时必须先确认当前目标属于哪个里程碑，避免提前实现后续阶段能力。

## 里程碑边界

严格按 `docs/05-里程碑.md` 推进：

- M1：凭据读取
- M2：额度 API 客户端
- M3：菜单栏文字 + 调度器 + 缓存
- M4：总览 Popover，cost 仍为占位 `-`
- M5：JSONL 扫描、Pricing、聚合，才接真实 cost
- M6：主窗口统计 tab，并把 Settings scene 迁到普通 Window
- M7：设置 tab 全量生效
- M8：桌面悬浮窗
- M9：打磨和发布准备

不要在当前里程碑里顺手实现后续功能。例如：

- M4 不做 JSONL cost。
- M4 不迁移 Settings scene。
- M5 前不要引入真实 usage 聚合。
- M6 前不要创建主窗口 TabView。
- M7 前不要把设置项完整接到运行时。

## 协作规则

- 未经明确要求“修改、实现、修复”等，不要擅自改代码。默认先分析、定位问题、给方案和风险。
- 如果需求、边界、验收标准或影响范围不明确，并且会影响实现或引入风险，先确认。
- 对大型项目，不要主动运行 `build`、`lint`、`type-check`、`test`、`dev` 等耗时或大量输出命令。确实需要时，先说明原因并询问。
- 修改前先读现有实现、项目规则、公共组件、公共规范和相关文档。
- 遵循当前项目风格，不做无关重构。
- 只改和当前任务直接相关的文件。不要顺手整理无关代码、格式或项目结构。
- 工作区可能已有用户改动。不要回滚、覆盖或清理非本次任务产生的改动。

## 实现约束

- UI 优先使用 SwiftUI / AppKit 系统默认控件和 macOS 原生风格。
- 视觉规则以 `docs/03-设计风格.md` 为准：
  - Codex 使用石墨灰识别色。
  - Claude Code 使用桃橙识别色。
  - 剩余额度为 `0` 时用红色。
  - 剩余额度小于 `20%` 时用橙色。
  - 数字使用 `.monospacedDigit()`。
  - 图标优先使用 SF Symbols。
- 不自造大面积背景、玻璃阴影、Web 风格控件或无关装饰。
- 菜单栏和 Popover 中 Codex 永远排在 Claude 前。
- 网络请求失败时保留已有快照，不要清空可展示数据。
- 429 后必须遵守现有退避策略，不要为了手动刷新绕过限流。
- M5 前本地 cost 只能显示占位，不能伪造。

## 常用验证方式

按任务风险选择最小验证手段：

- 小范围 UI 改动：先做静态检查和代码审阅。
- 需要编译确认时，先说明原因，再询问是否运行 Xcode 构建。
- 需要手动验收时，说明在 Xcode 中打开 `ccbar.xcodeproj` 并运行 App，按里程碑验收项检查。

不要为了验证一个小改动主动跑完整构建、完整测试或启动开发服务，除非用户明确要求。
